# 架构说明

## 整体架构

`vn-engine` 是本地优先的视觉小说引擎与编辑器 monorepo。工程分为应用层和核心包层：

- `apps/editor` 是编辑器界面，负责展示和修改工程数据。
- `apps/player` 是播放器，负责读取工程数据、驱动运行时并承载 PixiJS 舞台。
- `apps/desktop` 未来作为 Tauri 桌面壳，只提供本地文件能力。
- `packages/vn-schema` 定义工程数据结构和校验规则。
- `packages/vn-core` 执行剧情解释，不依赖 UI 或渲染框架。
- `packages/vn-project` 提供项目序列化、反序列化和深拷贝。
- `packages/vn-renderer-pixi` 接收 `RuntimeSnapshot` 和 `VNProject`，负责 PixiJS 画面渲染。
- `packages/vn-audio` 未来抽象音频播放。
- `packages/vn-ui-runtime` 未来沉淀运行时 UI 逻辑。
- `packages/vn-compiler` 未来可选实现脚本文法转换。

## 为什么 schema、core、renderer、editor 要分离

`schema` 是数据契约，`core` 是解释器，`renderer` 是画面适配，`editor` 是数据编辑界面。分离后，运行时可以被播放器、编辑器预览和测试工具共同复用，不会被 Vue 页面、PixiJS 或桌面壳绑定。

## 为什么数据是中心

视觉小说工程的核心资产是结构化数据：素材、角色、表情、脚本、节点、变量和跳转关系。只要数据结构稳定，编辑器、播放器、桌面壳和导出工具都可以围绕同一份工程数据演进。

## 为什么编辑器只是修改数据

编辑器不应该成为业务逻辑中心。它的职责是展示数据、触发操作、调用 service 或 store 修改数据。节点编辑、素材编辑、角色编辑、导入导出和运行预览都放在独立服务或核心包中，便于后续替换 UI 或接入 Tauri。

## 为什么播放器只是读取数据

播放器只负责读取项目数据、驱动 `VNRuntime`、把 `RuntimeSnapshot` 交给渲染器显示。它不保存工程，不编辑节点，也不直接解析底层文件结构。

## 为什么 renderer 只负责显示

`vn-renderer-pixi` 不持有 `VNRuntime`，也不调用 `next` 或 `choose`。它只根据当前快照和工程资源渲染背景、角色、对话和选项。选项点击通过回调交给播放器应用层，再由应用层调用运行时。

## 为什么 Tauri 只负责本地能力

Tauri 的职责是打开文件夹、读写文件、导入素材、导出游戏包和调用系统文件选择器。剧情解释、数据校验、素材引用规则和编辑器业务逻辑应留在 TypeScript 核心包和服务中，避免桌面壳绑定核心能力。

## 为什么导入导出先做 Web JSON

当前编辑器通过浏览器能力导入和下载 JSON 文件，序列化规则集中在 `packages/vn-project`。这样可以先验证项目数据保存与恢复闭环，避免过早把工程格式绑定到 Tauri 文件系统。

## 为什么素材库先做元数据

当前只管理素材 id、名称、类型、路径和说明，不处理真实二进制文件。播放器会尝试按路径加载图片，失败时使用占位图形。后续再接真实 Tauri 文件系统和素材导入。

## 为什么编辑器预览仍是文字化调试预览

本轮 PixiJS 渲染只接入正式播放器。编辑器底部预览继续展示运行时快照和资源解析结果，避免把复杂渲染生命周期塞进编辑器布局。

## 为什么不做节点图编辑器

节点图会引入拖拽、布局、连线、视口、选择状态和大量交互细节，容易掩盖更重要的数据结构和运行时稳定性。当前阶段先用节点列表验证数据和运行时。

## 为什么不做自定义脚本语言

自定义脚本语言需要 parser、错误定位、语法设计和编译产物映射。当前统一使用结构化 JSON，可以更快稳定 `StoryNode` 数据模型和运行时行为。

## 为什么 Element Plus 只用于编辑器

Element Plus 适合编辑器中的表单、树、列表、弹窗和面板，但不应该污染核心包、renderer 或播放器。核心包必须 UI 解耦，播放器也不应该呈现后台管理风格。
