# 架构说明

## 整体架构

`vn-engine` 是本地优先的视觉小说引擎与编辑器 monorepo。工程分为应用层和核心包层：

- `apps/editor` 是编辑器界面，负责修改工程数据。
- `apps/player` 是播放器，负责读取工程数据并展示运行时快照。
- `apps/desktop` 未来作为 Tauri 桌面壳，只提供本地文件能力。
- `packages/vn-schema` 定义工程数据结构。
- `packages/vn-core` 执行剧情解释，不依赖任何 UI 或渲染框架。
- `packages/vn-renderer-pixi` 未来适配 PixiJS 渲染。
- `packages/vn-project` 未来抽象本地工程读写。
- `packages/vn-audio` 未来抽象音频播放。
- `packages/vn-ui-runtime` 未来沉淀运行时 UI 逻辑。
- `packages/vn-compiler` 未来可选实现脚本文法转换。

## 为什么 schema、core、renderer、editor 要分离

`schema` 是数据契约，`core` 是解释器，`renderer` 是画面适配，`editor` 是数据编辑界面。分离后，运行时可以被播放器、编辑器预览、测试工具共同复用，不会被 Vue 页面或具体渲染实现绑定。

## 为什么数据是中心

视觉小说工程的核心资产是结构化数据：素材、角色、脚本、节点、变量和跳转关系。只要数据结构稳定，编辑器、播放器、桌面壳和导出工具都可以围绕同一份工程数据演进。

## 为什么编辑器只是修改数据

编辑器不应该成为业务逻辑中心。它的职责是展示数据、触发操作、调用 service 或 store 修改数据。这样未来替换 UI、接入 Tauri 或添加导出流程时，不需要重写核心逻辑。

## 为什么播放器只是读取数据

播放器只负责读取项目数据、驱动 `VNRuntime`、展示 `RuntimeSnapshot`。它不负责保存工程、不负责编辑节点，也不直接解析底层文件结构。

## 为什么 Tauri 只负责本地能力

Tauri 的职责是打开文件夹、读写文件、导入素材、导出游戏包和调用系统文件选择器。它不应该承载剧情解释、数据校验或编辑器业务逻辑，避免桌面壳绑死核心能力。

## 为什么第一阶段不做节点图编辑器

节点图会引入拖拽、布局、连线、视口、选择状态和大量交互细节，容易掩盖第一阶段最重要的数据结构和运行时稳定性。第一阶段先用节点列表验证数据和运行时。

## 为什么第一阶段不做自定义脚本语言

自定义脚本语言需要 parser、错误定位、语法设计和编译产物映射。第一阶段统一使用结构化 JSON，可以更快稳定 `StoryNode` 数据模型和运行时行为。

## 为什么 Element Plus 只用于编辑器

Element Plus 适合编辑器中的表单、树、列表、弹窗和面板，但不适合污染核心包或播放器。核心包必须与 UI 解耦，播放器也不应呈现后台管理风格。
